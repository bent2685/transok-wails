# 1. 共享文件夹采用请求时实时读盘

Date: 2026-05-30

## Status

Accepted

## Context

现有分享功能里，**分享清单（Share List）是一份快照**：分享方添加文件/文本时，文件信息被写死进 `storage.json`，被分享方通过 `POST /share/list` 一次性取走整份清单。这对单文件/纯文本足够，因为它们的内容在添加那一刻就固定了。

新需求要支持把**文件夹**加入分享清单，且要求"文件夹内容随磁盘实际变更动态变化"——分享方在文件夹里增删文件，被分享方刷新即可看到最新内容。

这与快照模型直接冲突：若沿用快照，添加文件夹时就得把当时的目录树写进清单，之后磁盘变更不会反映出来，违背"动态"要求。备选项还有"后台 fsnotify 监听 + 推送"和"定时轮询重扫快照"，但前者要引入文件监听与推送通道（WebSocket/SSE），后者有延迟且大目录反复全量扫描浪费。

## Decision

共享文件夹**不进入分享清单快照**。分享清单里只保存文件夹的根节点（`Type: "folder"`，`Path` 为目录绝对路径）。被分享方每次进入某个目录时，后端在请求那一刻 `os.ReadDir` 真实目录并返回当前内容。

为此新增浏览接口 `GET /share/browse`（实时列目录）与打包接口 `GET /download/zip`（流式打包某层目录）。请求**不接受客户端传入的绝对路径**，而是传 `folderId`（分享清单里某个 `Type:"folder"` 项的 Id）+ 相对子路径；后端用 Id 查出根绝对路径，再与相对子路径拼接。拼接结果经 `filepath.Clean` 与符号链接解析后，必须仍落在该根之内，否则一律拒绝；列目录时过滤以 `.` 开头的隐藏项。单文件下载同理改为按 folderId+相对路径授权（散文件仍走原有精确匹配）。

## Consequences

- **天然动态**：文件夹增删改在被分享方下次浏览时即时可见，无需监听或轮询，也不占内存。
- **行为不一致需解释**：同一份分享清单里，文件/文本是快照、文件夹是实时读盘——后来者会困惑为何两类项行为不同，本 ADR 即为此存在。
- **安全面扩大**：文件夹内的下载授权不再是"白名单精确匹配"，而是"folderId 定位根 + 相对路径拼接 + 根包含判断"，路径穿越与符号链接逃逸成为必须严防的攻击面；客户端永不传绝对路径，真实磁盘路径不外泄。
- **打包下载受限**：流式 zip 无法预知总大小、不支持 Range/断点续传，dash 端只能走原生 `<a>` 下载，不纳入现有分片并发进度管理。
- **dash 保持无路由**：下载页无 react-router，详情查看用 URL hash（`#item=<path>`）驱动 `DetailDialog` 弹窗。文件夹浏览复用同一 hash 机制（如 `#folder=<根>&sub=<相对路径>`），天然获得浏览器后退/前进与面包屑跳转，不引入路由库。
- **captcha 一致覆盖**：新增的 `/share/browse` 放入 `/share` 路由组、自动继承 captcha 中间件；`/download/zip` 复用 `/download/index` 的 captcha 校验。与现有文件下载同标准，不留绕过口。
- **失效语义对齐**：根目录被删/移走时，与现有文件一致地从清单过滤、浏览中报错退回，不引入占位机制。
